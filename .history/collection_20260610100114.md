1. # What is Collection Framework?

The Collection Framework is a unified architecture for representing and manipulating collections of objects in Java.

**Key Points:**
- Provides reusable data structures for storing and organizing objects
- Part of the java.util package
- Consists of interfaces (contracts for collections), implementations (concrete classes), and algorithms (methods for searching, sorting, etc.)
- Improves code quality by allowing developers to focus on business logic rather than implementing data structures
- Provides type-safe collections (Generics support)
- Offers high-performance implementations optimized for different use cases
- Reduces coding effort and increases code reusability
- Supports various collection types: List, Set, Queue, Map, and Deque

------------------------------------------------

2. What are the main interfaces in Collection Framework?

The Collection Framework is built on several core interfaces:

**List Interface:**
- Ordered, indexed collection that allows duplicates
- Maintains insertion order
- Provides positional access (get, set, add at index)
- Implementations: ArrayList, LinkedList, Vector, Stack

**Set Interface:**
- Unordered collection that does NOT allow duplicates
- Each element is unique
- Does not have indexing
- Implementations: HashSet, TreeSet, LinkedHashSet

**Queue Interface:**
- Follows FIFO (First-In-First-Out) principle
- Elements added at rear, removed from front
- Used for ordered processing of elements
- Implementations: PriorityQueue, ArrayDeque, LinkedList

**Map Interface:**
- Stores key-value pairs
- Keys are unique, values can be duplicated
- Does not extend Collection interface
- Implementations: HashMap, TreeMap, Hashtable, ConcurrentHashMap

**Deque Interface:**
- Double-ended queue allowing insertion/removal from both ends
- Can be used as both Queue and Stack
- Implementations: ArrayDeque, LinkedList

------------------------------------------------

3. # Difference between List and Set?

**List:**
- Maintains insertion order of elements
- Allows duplicate elements to be stored
- Indexed, meaning each element has a position (0-based)
- Positional access: get(), set(), add(index), remove(index)
- Better for accessing elements by position
- Examples: ArrayList, LinkedList, Vector
- Performance: ArrayList is O(1) for access, O(n) for insertion/deletion

**Set:**
- Does NOT allow duplicate elements
- May or may not maintain insertion order depending on implementation
- No indexing available
- No positional access methods
- Better for checking membership and unique collection handling
- Examples: HashSet (unordered), TreeSet (sorted), LinkedHashSet (insertion-ordered)
- Generally faster for membership testing (O(1) for HashSet)

**When to use:**
- Use List when you need ordered data with possible duplicates
- Use Set when you require unique elements
------------------------------------------------

4. # Difference between ArrayList and LinkedList?

**ArrayList:**
- Backed by a dynamic array (resizable array)
- Elements are stored in contiguous memory
- Excellent for random access: O(1) time complexity
- Slow for insertion/deletion in middle: O(n) time complexity
- Uses more memory if not fully filled (due to capacity > size)
- Synchronized version: Vector
- Better for frequent read operations
- Memory efficient for read-heavy workloads

**LinkedList:**
- Backed by a doubly linked list
- Elements are scattered in memory, connected via references
- Slow for random access: O(n) time complexity
- Fast for insertion/deletion at ends: O(1) time complexity
- Insertion/deletion in middle: O(n) time complexity
- Extra memory for storing node references (next and previous pointers)
- Better for frequent insertion/deletion operations
- Also implements Queue and Deque interfaces

**Performance Comparison:**
- Access: ArrayList O(1) vs LinkedList O(n)
- Insert/Delete at end: ArrayList O(1) vs LinkedList O(1)
- Insert/Delete in middle: ArrayList O(n) vs LinkedList O(n)
- Insert/Delete at beginning: ArrayList O(n) vs LinkedList O(1)

------------------------------------------------

5. # Difference between ArrayList and Vector?

**ArrayList:**
- Not synchronized (not thread-safe)
- Faster performance in single-threaded scenarios
- Better choice for modern applications
- Resizing factor: 50% (capacity increases by 50%)
- No legacy API compatibility concerns
- Introduced in Java 1.2
- Preferred for new code

**Vector:**
- Synchronized (thread-safe)
- Every method is synchronized, causing performance overhead
- Slower due to synchronization locks
- Resizing factor: 100% (capacity doubles)
- Legacy class from Java 1.0
- Better for multi-threaded environments (but CopyOnWriteArrayList is preferred now)
- Backward compatibility with old code
- Considered legacy; use ArrayList or CopyOnWriteArrayList instead

**When to use:**
- Use ArrayList for single-threaded or non-critical scenarios
- Use CopyOnWriteArrayList for thread-safe operations instead of Vector
- Vector should only be used for legacy code compatibility

**Note:** Both are indexed, maintain insertion order, and allow duplicates

6. What is HashSet?

HashSet is a collection class that implements the Set interface and stores unique elements.

**Key Characteristics:**
- Part of java.util package
- Does NOT allow duplicate elements
- Allows one null value
- Unordered collection (no insertion order guarantee)
- Uses HashMap internally for storage
- Each HashSet is backed by a HashMap instance
- Not synchronized (not thread-safe)
- Better performance: O(1) average case for add, remove, contains operations

**Internal Working:**
- Elements are stored as keys in an internal HashMap
- A dummy value is associated with each key
- Uses hash code to determine bucket location
- Collision handled using linked list or tree (Java 8+)

**Important Methods:**
- add(E e): Adds element if not present
- contains(Object o): Checks if element exists
- remove(Object o): Removes element
- size(): Returns number of elements

**When to use:**
- When you need a unique collection without duplicates
- When order doesn't matter
- For fast lookup and membership testing

**Thread Safety:**
- Use Collections.synchronizedSet() for thread-safe variant
- Or use ConcurrentHashMap-based alternatives

------------------------------------------------

7. # What is TreeSet?
TreeSet is a Set implementation that stores elements in a sorted order using a Red-Black Tree data structure.

**Key Characteristics:**
- Implements NavigableSet interface (extends SortedSet)
- Stores elements in sorted order
- Does NOT allow duplicate elements
- Does NOT allow null values (throws NullPointerException)
- Not synchronized (not thread-safe)
- Slower than HashSet due to sorting overhead
- Performance: O(log n) for add, remove, contains operations

**Internal Implementation:**
- Uses a Red-Black Tree (self-balancing binary search tree)
- Maintains elements in natural ordering or custom ordering
- Nodes are automatically balanced for optimal performance

**Sorting Order:**
- Natural order: Elements implement Comparable interface
- Custom order: Provide Comparator in constructor
- Elements must be comparable; otherwise ClassCastException occurs

**Important Methods:**
- add(E e): Adds element in sorted order
- first(): Returns smallest element
- last(): Returns largest element
- headSet(E toElement): Returns elements less than given element
- tailSet(E fromElement): Returns elements greater than or equal to given element
- subSet(E from, E to): Returns range of elements

**When to use:**
- When you need unique elements in sorted order
- When you need to perform range queries
- When order matters and performance overhead is acceptable

8. What is HashMap?

HashMap is a key-value pair data structure that implements the Map interface.

**Key Characteristics:**
- Stores unordered key-value pairs
- Allows ONE null key and MULTIPLE null values
- Not synchronized (not thread-safe)
- Implements Cloneable and Serializable interfaces
- Performance: O(1) average time complexity for get, put, remove
- Capacity can be resized dynamically
- Uses hash function to map keys to buckets

**Internal Working:**
- Uses an array of buckets (buckets[] or Entry[] array)
- Each bucket can store multiple key-value pairs (as nodes in linked list or tree)
- Hash code of key determines bucket index: index = hashCode % capacity
- Collision handling: Uses chaining (linked list) or tree (Java 8+)

**Important Characteristics:**
- Default initial capacity: 16
- Default load factor: 0.75
- Resizing occurs when size > capacity × load factor
- Does NOT maintain insertion order (use LinkedHashMap for that)
- Iterating order is unpredictable

**Important Methods:**
- put(K, V): Adds or updates key-value pair
- get(K): Retrieves value for given key
- remove(K): Removes key-value pair
- containsKey(K): Checks if key exists
- containsValue(V): Checks if value exists
- keySet(), values(), entrySet(): Get collections of keys, values, and entries

**Thread Safety:**
- Use Collections.synchronizedMap() for synchronized version
- Or use ConcurrentHashMap for better concurrent access performance

------------------------------------------------

9. Difference between HashMap and Hashtable?

**HashMap:**
- Not synchronized (not thread-safe)
- Better performance in single-threaded environments
- Allows ONE null key and multiple null values
- Introduced in Java 1.2 (as part of Collections framework)
- Legacy class but preferred over Hashtable
- Faster due to no synchronization overhead
- Iterator is fail-fast (throws ConcurrentModificationException)
- Default capacity: 16, Load factor: 0.75
- Resizes by 50% when load factor threshold is exceeded

**Hashtable:**
- Synchronized (thread-safe)
- Every method is synchronized, causing performance overhead
- Does NOT allow null keys or null values
- Introduced in Java 1.0 (legacy class)
- Slower due to synchronization locks
- Should NOT be used in new code
- Enumerator is NOT fail-fast (doesn't throw exception on modification)
- Default capacity: 11, Load factor: 0.75
- Resizes by 100% (doubles) when load factor threshold is exceeded

**Comparison Table:**

| Feature     | HashMap     | Hashtable   |
|--------- ---|---------    |-------------|
| Thread-safe | No          |Yes          |             
| Null key    | Allowed (1) | Not allowed |
| Null value  | Allowed     | Not allowed |
| Performance | Faster      | Slower      |
| Legacy      | No          | Yes         |
| Iteration   | Fail-fast   | Not fail-fast |
| Recommendation | Use this | Use ConcurrentHashMap instead |

**When to use:**
- Use HashMap for single-threaded or non-critical scenarios
- Use ConcurrentHashMap for multi-threaded environments
- Avoid Hashtable completely (only for legacy code compatibility)

10. Difference between HashMap and ConcurrentHashMap?

**HashMap:**
- Not thread-safe (not synchronized)
- Allows ONE null key and multiple null values
- No locking mechanism
- Fast performance in single-threaded scenarios
- Iterator is fail-fast (throws ConcurrentModificationException on modification)
- Better for non-concurrent applications
- Simple synchronization: Collections.synchronizedMap()
- All methods perform at normal speed

**ConcurrentHashMap:**
- Thread-safe and designed for concurrent access
- Does NOT allow null keys or null values
- Uses segment-based locking (bucket-level locking) for fine-grained synchronization
- Better performance in multi-threaded scenarios
- Iterator is fail-safe (doesn't throw ConcurrentModificationException)
- Allows multiple threads to access different segments simultaneously
- More efficient than synchronized HashMap for concurrent access
- Introduced in Java 1.5

**Performance Comparison:**
- Single-threaded: HashMap faster
- Multi-threaded: ConcurrentHashMap much faster due to segment locking
- ConcurrentHashMap allows 16 concurrent threads by default (segment count)

**Internal Locking Strategy:**
- HashMap: No locking
- ConcurrentHashMap: Bucket-level locks (segment locks in Java 7, node-level locks in Java 8+)
- Multiple threads can read/write different buckets simultaneously

**When to use:**
- Use HashMap when: Single-threaded or not performance critical
- Use ConcurrentHashMap when: Multi-threaded access with high concurrency
- DON'T use Collections.synchronizedMap(HashMap) - ConcurrentHashMap is better

**Null Handling:**
- HashMap: Allows null key and null values
- ConcurrentHashMap: Doesn't allow null (raises NullPointerException)

11. # What is Load Factor?

Load Factor is a measure used by HashMap to determine when to resize (increase capacity).

**Definition:**
- It's a ratio that indicates how full the HashMap can get before resizing
- Formula: Load Factor = (Number of entries) / (Current Capacity)
- Resizing occurs when: size > capacity × load factor

**Default Value:**
- Default load factor = 0.75 (75%)
- This provides good balance between time and space efficiency

**How it works:**
- Load factor of 0.75 means resize when 75% of buckets are filled
- For default capacity of 16: Resize happens when size > 12 elements
- After resizing, capacity becomes 32, and resizing threshold becomes 24

**Effect of Different Load Factors:**

**Low Load Factor (0.25):**
- More frequent resizing
- Less chance of hash collisions
- More memory wasted (sparse hashmap)
- Faster lookup operations

**High Load Factor (0.9):**
- Less frequent resizing
- More memory efficient
- Higher chance of hash collisions
- Slower lookup operations due to collision handling

**Trade-offs:**
- Lower load factor = more memory, faster access
- Higher load factor = less memory, slower access
- 0.75 is optimal balance for most scenarios

**Custom Load Factor:**
- Can be specified in HashMap constructor
- Example: `new HashMap<>(16, 0.5)` - capacity 16, load factor 0.5

**Performance Impact:**
- Too low: Wastes memory, excessive resizing overhead
- Too high: More collisions, slower performance
- 0.75: Good default recommended by Java designers

12. What is Initial Capacity?

Initial Capacity is the number of buckets (slots) in a HashMap when it is first created.

**Definition:**
- The size of the internal bucket array (capacity) at initialization
- Capacity is ALWAYS a power of 2 (16, 32, 64, 128, etc.)
- Different from size: size is number of entries, capacity is bucket count

**Default Initial Capacity:**
- Default = 16 buckets
- This is the default capacity for `new HashMap()`

**Why Power of 2?**
- Allows faster bitwise operations to calculate bucket index
- Formula: index = hashCode & (capacity - 1) is equivalent to hashCode % capacity
- Bitwise AND operation is much faster than modulo operation
- Example: capacity = 16 (binary 10000), so capacity-1 = 15 (binary 1111)

**Custom Initial Capacity:**
- Can be specified in constructor: `new HashMap<>(32)` - capacity 32
- If specified capacity is not power of 2, Java rounds it up
- Example: `new HashMap<>(15)` becomes capacity 16
- Example: `new HashMap<>(20)` becomes capacity 32

**Performance Implications:**
- Choosing right initial capacity avoids multiple resizing operations
- Each resize operation is O(n) and expensive
- If you know approximate size, set initial capacity accordingly
- Formula: capacity = (expected_size / 0.75) + 1
- Example: For 1000 elements, set capacity to 1334+

**Memory Considerations:**
- Initial capacity reserves memory even if not all buckets are used
- Too large initial capacity wastes memory
- Too small initial capacity causes frequent resizing

**Example:**
```java
// Default capacity = 16
Map<String, String> map1 = new HashMap<>();

// Custom capacity = 64
Map<String, String> map2 = new HashMap<>(64);

// With load factor (capacity = 32, load factor = 0.5)
Map<String, String> map3 = new HashMap<>(32, 0.5f);
```

13. # What is Hash Collision?

Hash Collision occurs when two different keys produce the same hash code and map to the same bucket in a HashMap.

**Definition & Example:**
- Multiple keys with different values hash to the same bucket index
- Example: Keys "AB" and "BA" might have same hash code due to algorithm properties
- Hash collisions are inevitable in any hash table implementation

**Why Collisions Occur:**
- Hash codes are unlimited (Integer.MAX_VALUE range)
- HashMap buckets are limited (initial capacity 16)
- Multiple keys can hash to same value (pigeonhole principle)
- Poor hash function increases collision probability

**Impact on Performance:**
- Single key: O(1) constant time lookup
- Hash collision: Search through multiple entries in same bucket
- Many collisions: Performance degrades to O(n) linear time
- Load factor controls collision probability

**Collision Handling Methods:**

**1. Chaining (Java 7 and before):**
- Each bucket stores a linked list of entries (nodes)
- Collision: New entry added to linked list
- Search: Traverse linked list to find correct entry
- Time complexity: O(n) in worst case (all keys in one bucket)

**2. Tree Hashing (Java 8+):**
- When linked list size exceeds threshold (8 nodes)
- Bucket converts from linked list to Red-Black Tree
- Search: Binary search in tree gives O(log n) time
- Threshold back to linked list when size drops below 6
- Significantly improves collision handling performance

**Factors Affecting Collisions:**
- Load factor: Higher load factor = more collisions
- Hash function quality: Poor function causes more collisions
- Key distribution: Similarly distributed keys increase collisions
- Capacity: Small capacity increases collision probability

**Practical Impact:**
- Few collisions (0.75 load factor): O(1) average performance
- Many collisions: O(n) worst case performance
- Java 8+ trees reduce impact of collisions

14. How does HashMap handle collision?

HashMap uses multiple collision-handling strategies to manage cases where different keys hash to the same bucket.

**Collision Handling Methods:**

**Method 1: Chaining (Java 7 and earlier):**
- Each bucket stores a linked list (chain) of key-value pairs
- When collision occurs: New entry appended to linked list
- Lookup process:
  1. Calculate bucket index using hash code
  2. Traverse linked list in that bucket
  3. Compare keys using equals() method
  4. Return value if key matches
- Time complexity: O(1) average, O(n) worst case
- Memory: Extra space for node references

**Method 2: Tree Hashing (Java 8+):**
- Hybrid approach combining chaining and red-black tree
- Step 1: Initially stores collisions as linked list (chain)
- Step 2: When chain size exceeds TREEIFY_THRESHOLD (default = 8):
  - Bucket converts from linked list to Red-Black Tree
  - Nodes are compared using Comparable interface
- Step 3: When chain size drops below UNTREEIFY_THRESHOLD (default = 6):
  - Binary tree reverts to linked list
- Benefits:
  - Linked list: O(k) where k is chain length
  - Red-Black Tree: O(log k) much faster for long chains
  - Prevents performance degradation in worst case

**Comparison of Strategies:**

| Aspect | Chaining | Tree Hashing |
|--------|----------|---------------|
| Best Case | O(1) | O(1) |
| Average Case | O(1) | O(1) |
| Worst Case | O(n) | O(log n) |
| Memory | Low | Higher |
| Java Version | Pre-8 | 8+ |
| Collision cost | Linear | Logarithmic |

**Example with Tree Conversion:**
```java
HashMap<String, Integer> map = new HashMap<>();
// Add entries with same hash code
// After 8 collisions in one bucket
// -> Linked list converts to Red-Black Tree
// -> Access becomes O(log n) instead of O(n)
```

**Best Practices to Minimize Collisions:**
- Use good hashCode() implementation
- Ensure equals() and hashCode() are consistent
- Set appropriate initial capacity
- Maintain reasonable load factor (0.75)

15. What is fail-fast iterator?

A fail-fast iterator is an iterator that detects modification of a collection during iteration and immediately throws ConcurrentModificationException.

**Definition:**
- Detects structural modifications (add/remove/clear) to collection during iteration
- Throws ConcurrentModificationException when detecting modification
- "Fails fast" = stops immediately instead of allowing incorrect behavior
- Fail-fail approach prevents subtle bugs

**How it Works:**
- Iterator maintains an internal counter called modCount (modification count)
- During iteration, iterator checks if modCount has changed
- If changed: Throws ConcurrentModificationException
- Exception thrown at next iteration attempt (next() call)

**Important Behavior:**
- The exception is thrown BEFORE invalid data is accessed
- Iterator detects both external and internal modifications
- Each collection tracks modification count separately

**Collections with Fail-Fast Iterators:**
- ArrayList
- LinkedList
- HashSet
- TreeSet
- HashMap
- TreeMap


**Example Code:**
```java
List<String> list = new ArrayList<>();
list.add("A");
list.add("B");
list.add("C");

Iterator<String> iterator = list.iterator();
while(iterator.hasNext()) {
    System.out.println(iterator.next());
    list.add("D");  // Modification during iteration
    // Result: ConcurrentModificationException thrown
}
```

**How to Handle Safely:**

**Option 1: Use Iterator.remove()**
```java
Iterator<String> iterator = list.iterator();
while(iterator.hasNext()) {
    String item = iterator.next();
    if (item.equals("B")) {
        iterator.remove();  // Safe removal
    }
}
```

**Option 2: Use removeIf()**
```java
list.removeIf(item -> item.equals("B"));  // Safe in Java 8+
```

**Option 3: Create copy for iteration**
```java
for (String item : new ArrayList<>(list)) {
    list.remove(item);  // Safe as iterating on copy
}
```

**Why Fail-Fast?**
- Prevents data corruption from concurrent modifications
- Makes bugs visible immediately (better than silent failures)
- Forces developers to use proper synchronization

**Limitations:**
- Not thread-safe; just detects same-thread modifications
- Multiple threads can still cause issues
- For multi-threaded access, use synchronized collections or ConcurrentHashMap

16. # What is fail-safe iterator?

A fail-safe iterator is an iterator that works on a copy of the collection and does NOT throw ConcurrentModificationException if the underlying collection is modified.

**Definition:**
- Iterates over a snapshot/copy of collection at the time iterator was created
- Modifications to original collection are not detected
- Does NOT throw ConcurrentModificationException
- "Fail-safe" = continues working even if collection changes
- Safe for concurrent access and modifications

**How it Works:**
- Creates a shallow copy of collection when iterator is created
- Iterator traverses the copy, not the original collection
- Original collection can be modified without affecting iterator
- Iterator never sees the modifications made after iteration started

**Trade-offs:**
- **Advantage:** Safe for concurrent modifications
- **Disadvantage:** Memory overhead due to copying
- **Disadvantage:** Iterator sees stale data (pre-modification state)
- **Disadvantage:** Additional CPU cost for copying

**Collections with Fail-Safe Iterators:**
- ConcurrentHashMap
- CopyOnWriteArrayList
- CopyOnWriteArraySet
- Collections created by Collections.synchronizedCollection()

**Example with ConcurrentHashMap:**
```java
Map<String, Integer> map = new ConcurrentHashMap<>();
map.put("A", 1);
map.put("B", 2);
map.put("C", 3);

Iterator<String> iterator = map.keySet().iterator();
while(iterator.hasNext()) {
    System.out.println(iterator.next());
    map.put("D", 4);  // Safe - no exception thrown
}
```

**Example with CopyOnWriteArrayList:**
```java
List<String> list = new CopyOnWriteArrayList<>();
list.add("A");
list.add("B");

for (String item : list) {
    System.out.println(item);
    list.add("C");  // No exception, but new item not in iteration
}
```

**Comparison: Fail-Fast vs Fail-Safe:**

| Feature | Fail-Fast | Fail-Safe |
|---------|-----------|----------|
| Exception on modification | Yes | No |
| Thread-safe | No | Yes |
| Memory efficiency | High | Low (copy cost) |
| Data consistency | Always latest | Snapshot state |
| Implementation | ArrayList, HashMap | ConcurrentHashMap, CopyOnWriteArrayList |
| Performance | Fast | Slower (copy overhead) |
| Use case | Single-threaded | Multi-threaded |

**When to Use:**
- Use Fail-Safe when: Multiple threads access collection
- Use Fail-Safe when: Concurrent modifications are expected
- Use Fail-Safe when: Want to avoid exceptions during iteration
- Trade-off: Accept memory overhead and potential stale data

17. Difference between Iterator and ListIterator?

**Iterator:**
- Part of java.util.Iterator interface
- Works with all Collection types (List, Set, Queue)
- Allows traversal in forward direction only (unidirectional)
- Three main methods:
  - hasNext(): Checks if more elements exist
  - next(): Returns next element and moves cursor forward
  - remove(): Removes current element from underlying collection
- Cannot go backward through collection
- Cannot add elements during iteration
- Cannot retrieve index position

**ListIterator:**
- Extends Iterator interface (javs.util.ListIterator)
- Works only with List collections (ArrayList, LinkedList, etc.)
- Allows traversal in both forward and backward directions (bidirectional)
- All Iterator methods plus additional methods:
  - hasPrevious(): Checks if previous element exists
  - previous(): Returns previous element and moves cursor backward
  - nextIndex(): Returns index of next element
  - previousIndex(): Returns index of previous element
  - set(E e): Replaces current element
  - add(E e): Inserts element at current position
- Can traverse forward and backward
- Can add elements during iteration
- Can know current position (index)

**Detailed Comparison Table:**

-----------------------------------------------------------------
| Feature      | Iterator             | ListIterator            |
|--------------|--- ---------------   |-------------------------|
| Direction    | Forward only         | Both forward & backward |
| Interface    | Extends Collection   | Extends Iterator        |
| Collections  | All types            | Only List               |
| Methods count| 3                    |9                        |
| hasNext()    | Yes                  | Yes                     |
| next()       | Yes                  | Yes                     |
| hasPrevious()| No                   | Yes                     |
| previous()   | No                   | Yes                     |
| remove()     | Yes                  | Yes                     |
| set(E)       | No                   | Yes                     |
| add(E)       | No                   | Yes                     |
| Index access | No                   |Yes (next, previousIndex)|
| Performance  |O(1)LinkedList forward| O(n) LL backward        |
-----------------------------------------------------------------

**Iterator Example:**
```java
List<String> list = new ArrayList<>();
list.add("A");
list.add("B");
list.add("C");

Iterator<String> iterator = list.iterator();
while(iterator.hasNext()) {
    String item = iterator.next();
    if(item.equals("B")) {
        iterator.remove();  // Safe removal
    }
}
```

**ListIterator Example:**
```java
List<String> list = new ArrayList<>();
list.add("A");
list.add("B");
list.add("C");

ListIterator<String> listIterator = list.listIterator();
while(listIterator.hasNext()) {
    String item = listIterator.next();
    System.out.println(item);
}

// Now go backward
while(listIterator.hasPrevious()) {
    String item = listIterator.previous();
    System.out.println(item);
}

// Add element while iterating
listIterator.add("D");  // Safe with ListIterator
```

**When to Use:**
- Use Iterator: When you need simple forward iteration
- Use ListIterator: When you need bidirectional iteration or modification
- Use List.listIterator(): Get ListIterator from List collections

18. # What is Comparable?

Comparable is an interface used for natural sorting of objects based on their default ordering.

**Definition:**
- Part of java.lang.Comparable interface
- Used to define natural order of objects
- "Natural" = default or intended ordering
- Objects implementing Comparable can be sorted automatically

**Key Method:**
```java
public int compareTo(T o)
```

**Return Value Meaning:**
- Negative: This object is less than given object
- Zero (0): This object equals the given object
- Positive: This object is greater than given object

**How Comparable Works:**
- Class implements Comparable interface
- Override compareTo() method to define comparison logic
- Objects compared according to compareTo() implementation
- Used by sorting algorithms (Arrays.sort(), Collections.sort())

**Implementation Example:**
```java
class Student implements Comparable<Student> {
    String name;
    int marks;
    
    @Override
    public int compareTo(Student other) {
        // Sort by marks in ascending order
        return this.marks - other.marks;
    }
}

// Usage
List<Student> students = new ArrayList<>();
students.add(new Student("Alice", 85));
students.add(new Student("Bob", 75));
students.add(new Student("Charlie", 95));

Collections.sort(students);  // Uses compareTo()
```

**Natural Ordering Examples:**
- String: Alphabetical order (a < b < c ...)
- Integer: Numerical order (1 < 2 < 3 ...)
- Date: Chronological order

**Collections Using Comparable:**
- TreeSet: Sorts elements using compareTo()
- TreeMap: Sorts keys using compareTo()
- Collections.sort(): Sorts using compareTo()
- Arrays.sort(): Sorts using compareTo()

**Advantages:**
- Simple for single sorting criterion
- Class defines its own natural order
- No need for external Comparator
- Works with any collection needing sorted order

**Limitations:**
- Only one compareTo() per class
- Cannot change natural order for existing class
- To sort by different criteria, need Comparator

**Comparable vs Comparator:**
--------------------------------------------------------------------
| Aspect            | Comparable          | Comparator          |
|-------------------|-------------------- |------------------   |
| Interface location| java.lang           | java.util           |
| Method            | compareTo(T)        | compare(T, T)       |
| Implementation    |In class to be sorted| External/separate   |
| Sorting criteria  | One (natural order) | Multiple possible   |
| Mutability        | Modifies class      |No class modification|
| Flexibility       | Less flexible       | More flexible       |
--------------------------------------------------------------------
**When to Use:**
- Use Comparable: When there's an obvious natural order
- Use Comparator: When you need flexible sorting strategies

19. # What is Comparator?

Comparator is an interface used for custom sorting of objects with flexible comparison logic.

**Definition:**
- Part of java.util.Comparator interface
- Used to define custom ordering separate from the class
- Allows multiple different sorting strategies
- "Custom" = not dependent on natural order
- Provides flexible and reusable sorting logic

**Key Method:**
```java
public int compare(T o1, T o2)
```

**Return Value Meaning:**
- Negative: First object is less than second object
- Zero (0): Objects are equal
- Positive: First object is greater than second object

**How Comparator Works:**
- Create separate class implementing Comparator
- Override compare() method with custom comparison logic
- Pass Comparator instance to sorting methods
- Comparison logic defined externally from class

**Implementation Example:**
```java
class Student {
    String name;
    int marks;
    
    Student(String name, int marks) {
        this.name = name;
        this.marks = marks;
    }
}

// Comparator for sorting by marks
class MarkComparator implements Comparator<Student> {
    @Override
    public int compare(Student s1, Student s2) {
        return s1.marks - s2.marks;  // Ascending by marks
    }
}

// Comparator for sorting by name
class NameComparator implements Comparator<Student> {
    @Override
    public int compare(Student s1, Student s2) {
        return s1.name.compareTo(s2.name);  // Alphabetical
    }
}

// Usage
List<Student> students = new ArrayList<>();
students.add(new Student("Alice", 85));
students.add(new Student("Bob", 75));

// Sort by marks
Collections.sort(students, new MarkComparator());

// Sort by name
Collections.sort(students, new NameComparator());
```

**Anonymous Comparator (Java 7):**
```java
Collections.sort(students, new Comparator<Student>() {
    @Override
    public int compare(Student s1, Student s2) {
        return s1.marks - s2.marks;
    }
});
```

**Lambda Expression (Java 8+):**
```java
// Most concise way
Collections.sort(students, (s1, s2) -> s1.marks - s2.marks);

// Or using Comparator.comparing()
students.sort(Comparator.comparingInt(s -> s.marks));

// Descending order
students.sort(Comparator.comparingInt((Student s) -> s.marks).reversed());
```

**Key Differences from Comparable:**

| Aspect                  | Comparable | Comparator |
|-------------------------|-----------------------------|-------------------------------|
| Interface Package       | java.lang                   | java.util                     |
| Method                  | compareTo(T)                | compare(T, T)                 |
| Implementation Location | Inside class                | External/separate             |
| Number of strategies    | One per class               | Multiple per class            |
| Needs implementation    | Class modification required | No class change               |
| Parameters to sort()    | Collections.sort(list)      | Collections.sort(list, comparator) |
| Flexibility             | Fixed order                 | Can change dynamically        |

**Collections Supporting Comparator:**
- TreeSet: `new TreeSet<>(comparator)`
- TreeMap: `new TreeMap<>(comparator)`
- Collections.sort(): `Collections.sort(list, comparator)`
- PriorityQueue: `new PriorityQueue<>(comparator)`
- Arrays.sort(): `Arrays.sort(array, comparator)`

**Advantages of Comparator:**
- Multiple sorting strategies
- No need to modify original class
- Reusable for different classes
- Can sort third-party classes without modification
- More flexible than Comparable

**When to Use:**
- Use Comparator: When you need multiple sorting strategies
- Use Comparator: When you cannot modify the class
- Use Comparator: For complex sorting logic
- Use Comparator: When sorting third-party classes

20. # What is PriorityQueue?

PriorityQueue is a queue implementation where elements are processed based on their priority rather than FIFO order.

**Definition:**
- Part of java.util package
- Queue where elements are retrieved by priority
- Not a FIFO (First-In-First-Out) structure
- Based on binary min-heap data structure (by default)
- Implements Queue interface

**Key Characteristics:**
- Elements ordered by priority (natural order or custom Comparator)
- Head of queue = element with highest priority
- Lower values = higher priority (by default min-heap)
- Does NOT allow null elements
- Not synchronized (not thread-safe)
- Use PriorityBlockingQueue for thread-safety
- Time complexity: O(log n) for insertion and removal

**Default Ordering (Min-Heap):**
- Smallest element has highest priority
- "Smallest" based on natural ordering (Comparable) or Comparator
- Example: Numbers 3, 1, 4, 1, 5 → retrieved as 1, 1, 3, 4, 5

**Important Methods:**
- add(E e): Inserts element (throws exception if fails)
- offer(E e): Inserts element (returns false if fails)
- poll(): Removes and returns highest priority element
- peek(): Returns highest priority element without removing
- remove(): Removes and returns head (throws exception if empty)
- element(): Returns head (throws exception if empty)

**Example with Natural Ordering:**
```java
PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.add(5);
pq.add(1);
pq.add(3);

while(!pq.isEmpty()) {
    System.out.println(pq.poll());  // Output: 1, 3, 5
}
```

**Example with Custom Comparator (Max-Heap):**
```java
// Max-heap: Largest element first
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(
    (a, b) -> b - a  // Reverse order
);
maxHeap.add(5);
maxHeap.add(1);
maxHeap.add(3);

while(!maxHeap.isEmpty()) {
    System.out.println(maxHeap.poll());  // Output: 5, 3, 1
}
```

**Internal Structure - Binary Heap:**
- Array-based representation of binary tree
- Parent at index i
- Left child at index 2i+1
- Right child at index 2i+2
- Efficient for priority retrieval

**Performance Characteristics:**

| Operation            | Complexity |
|----------- ----------|------------|
| Insert (add/offer)   | O(log n)   |
| Delete (poll/remove) | O(log n)   |
| Peek (element/peek)  | O(1)       |
| Search (contains)    | O(n)       |
| Creation             | O(n)       |

**Common Use Cases:**
- Job scheduling (higher priority jobs first)
- Dijkstra's algorithm (shortest path)
- Huffman coding (tree construction)
- Patient queue in hospital (critical patients first)
- Bandwidth management (important traffic first)
- Task scheduling systems

**Example: Patient Priority Queue:**
```java
class Patient implements Comparable<Patient> {
    String name;
    int severity;  // 1=critical, 5=minor
    
    @Override
    public int compareTo(Patient other) {
        return this.severity - other.severity;  // Lower severity = higher priority
    }
}

PriorityQueue<Patient> queue = new PriorityQueue<>();
queue.add(new Patient("John", 3));
queue.add(new Patient("Alice", 1));  // Critical
queue.add(new Patient("Bob", 5));    // Minor

// Alice (critical) treated first, then John, then Bob
```

**Limitations:**
- Not ordered (internal structure is heap, not sorted)
- Using iterator doesn't guarantee order
- Slow search operation O(n)
- Not suitable for frequent rank queries

**Thread-Safe Alternative:**
- PriorityBlockingQueue: Thread-safe version
- Used in concurrent applications
- Blocking operations for producer-consumer pattern

21. # What is Deque?

Deque (Double-Ended Queue) is a queue data structure where elements can be inserted or removed from both ends.

**Definition:**
- Deque = Double-Ended Queue
- Part of java.util package
- Supports insertion and removal at both front and rear
- Can be used as both Queue (FIFO) and Stack (LIFO)
- More flexible than Queue or Stack alone

**Key Characteristics:**
- Can add/remove from both ends (efficient O(1) operations)
- FIFO operations at one end, LIFO at other end
- Two-way operations provide maximum flexibility
- Null elements allowed (depends on implementation)
- Implementations: ArrayDeque, LinkedList, ConcurrentLinkedDeque

**Important Methods:**

**Front-end operations (Queue-style):**
- addFirst(E e): Add to front, throws exception if fails
- addLast(E e): Add to rear, throws exception if fails
- removeFirst(): Remove from front, throws exception if empty
- removeLast(): Remove from rear, throws exception if empty
- getFirst(): Get front without removing, throws exception if empty
- getLast(): Get rear without removing, throws exception if empty

**Front-end operations (Safe versions):**
- offerFirst(E e): Add to front, returns false if fails
- offerLast(E e): Add to rear, returns false if fails
- pollFirst(): Remove from front, returns null if empty
- pollLast(): Remove from rear, returns null if empty
- peekFirst(): Get front without removing, returns null if empty
- peekLast(): Get rear without removing, returns null if empty

**As Queue (FIFO):**
- add()/offer(): Add to rear
- remove()/poll(): Remove from front
- element()/peek(): Get front

**As Stack (LIFO):**
- push(E e): Add to front
- pop(): Remove from front
- peek(): Get front

**Example Usage:**
```java
Deque<String> deque = new LinkedList<>();

// Add elements
deque.addFirst("A");  // Front
deque.addLast("B");   // Rear
deque.addLast("C");

// Deque now: [A, B, C]

// Remove from both ends
System.out.println(deque.removeFirst());  // A
System.out.println(deque.removeLast());   // C

// Remaining: [B]
```

**Using as Stack:**
```java
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);
stack.push(2);
stack.push(3);

while(!stack.isEmpty()) {
    System.out.println(stack.pop());  // 3, 2, 1 (LIFO)
}
```

**Using as Queue:**
```java
Deque<Integer> queue = new ArrayDeque<>();
queue.offer(1);
queue.offer(2);
queue.offer(3);

while(!queue.isEmpty()) {
    System.out.println(queue.poll());  // 1, 2, 3 (FIFO)
}
```

**Implementations Comparison:**
|-------------------------------------------------------------------------------|
| Implementation        | Thread-Safe | Null Allowed | Sorted | Performance     |
|-----------------------|-------------|--------------|--------|-----------------|
| ArrayDeque            | No          | No           | No     | O(1) amortized  |
| LinkedList            | No          | Yes          | No     | O(1)            |
| ConcurrentLinkedDeque | Yes         | No           | No     | O(1) concurrent |
|-------------------------------------------------------------------------------|
**Performance Characteristics:**
- ArrayDeque: Very efficient, preferred for most cases
- LinkedList: Also O(1) for all operations, but more memory
- Both much better than Queue/Stack for double-ended operations

**Advantages:**
- Single data structure can act as Queue or Stack
- Efficient operations at both ends
- Reduces code complexity (no need separate Stack class)
- Better performance than using List with index operations

**When to Use:**
- Use Deque: When you need operations at both ends
- Use ArrayDeque: For best performance as Stack or Queue
- Use LinkedList: Only if you need other List features
- Use ConcurrentLinkedDeque: For thread-safe operations

22. # What is BlockingQueue?
BlockingQueue is a thread-safe queue used in producer-consumer problems where threads can block if queue is full or empty.                                                                                                                              

**Definition:**
- Thread-safe queue that extends Queue interface
- Part of java.util.concurrent package
- Blocks thread operations based on queue state
- Designed for multi-threaded producer-consumer scenarios
- Part of Java Collections Framework for concurrency

**Key Characteristics:**
- Thread-safe: No external synchronization needed
- Blocking operations: Threads wait for queue state change
- Bounded capacity: Queue has fixed maximum size
- Null elements NOT allowed
- Used for coordinating multiple threads
- Fair or unfair waiting (depends on implementation)

**Blocking Methods:**

**Put operations (for Producers):**
- put(E e): Adds element, waits if queue is full
- offer(E e, long timeout, TimeUnit unit): Adds with timeout

**Take operations (for Consumers):**
- take(): Removes element, waits if queue is empty
- poll(long timeout, TimeUnit unit): Removes with timeout

**Non-blocking Methods:**
- add(E e): Throws exception if queue full
- offer(E e): Returns false if queue full
- remove(): Throws exception if queue empty
- poll(): Returns null if queue empty
- element(): Throws exception if queue empty
- peek(): Returns null if queue empty

**Main Implementations:**

**1. ArrayBlockingQueue:**
- Bounded queue (fixed capacity)
- Based on array
- Single lock for thread safety
- Fair/unfair options
- FIFO ordering

**2. LinkedBlockingQueue:**
- Unbounded queue (or optionally bounded)
- Based on linked nodes
- Two separate locks (more concurrent)
- Better performance than ArrayBlockingQueue
- FIFO ordering

**3. PriorityBlockingQueue:**
- Unbounded queue
- Elements ordered by priority
- Based on heap
- Thread-safe version of PriorityQueue

**4. SynchronousQueue:**
- Queue with capacity 0
- Producer waits for consumer to take
- No internal storage
- Used for handoff between threads

**5. DelayQueue:**
- Unbounded queue
- Elements have delays
- Elements can only be taken when delay expires
- Used for scheduled task execution

**Producer-Consumer Example:**
```java
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(5);

// Producer thread
Thread producer = new Thread(() -> {
    try {
        for (int i = 1; i <= 10; i++) {
            queue.put(i);           // Blocks if queue full
            System.out.println("Produced: " + i);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

// Consumer thread
Thread consumer = new Thread(() -> {
    try {
        while (true) {
            Integer item = queue.take();  // Blocks if queue empty
            System.out.println("Consumed: " + item);
            Thread.sleep(1000);  // Simulate processing
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

producer.start();
consumer.start();
```

**Behavior Flow:**

```
Producer tries to put():
- Queue not full → Element added immediately
- Queue is full → Producer thread BLOCKED until space available

Consumer tries to take():
- Queue not empty → Element removed immediately
- Queue is empty → Consumer thread BLOCKED until element available
```

**Performance Characteristics:**

| Operation | ArrayBlockingQueue | LinkedBlockingQueue | Time |
|-----------|--------------------|---------------------|------|
| put()     | Synchronized       | Two-lock            | O(1) |
| take()    | Synchronized       | Two-lock            | O(1) |
| Concurrency | Single lock      | Better              | Depends on impl |
| Memory    | Fixed              | Dynamic             | LinkedList more |

**Use Cases:**
- Producer-Consumer pattern
- Thread pool task queues
- Rate limiting
- Load balancing
- Work distribution
- Buffer between fast producer and slow consumer

**Example: Work Queue:**
```java
BlockingQueue<String> workQueue = new LinkedBlockingQueue<>();

// Multiple workers take tasks
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        while (true) {
            try {
                String task = workQueue.take();
                System.out.println("Processing: " + task);
            } catch (InterruptedException e) {
                break;
            }
        }
    }).start();
}

// Producer queues tasks
workQueue.put("Task1");
workQueue.put("Task2");
workQueue.put("Task3");
```

**Advantages:**
- Built-in thread synchronization
- Automatic blocking/waiting
- No complex lock management
- Producer-consumer coordination
- Back-pressure handling (queue full)

**Comparison with Queue:**
- Regular Queue: Not thread-safe, no blocking
- BlockingQueue: Thread-safe, blocking operations
- BlockingQueue: Designed for concurrent scenarios

23. # What is CopyOnWriteArrayList?

CopyOnWriteArrayList is a thread-safe variant of ArrayList where modification operations create a new copy of the underlying array.

**Definition:**
- Thread-safe List implementation
- Part of java.util.concurrent package
- Uses Copy-On-Write (COW) pattern
- Optimized for read-heavy workloads
- All threads see consistent state

**Key Characteristics:**
- Thread-safe without synchronization (no locks needed)
- Read operations do NOT acquire locks
- Write operations (add, remove, set) acquire locks
- Immutable snapshot taken during iteration
- Iterator is fail-safe (safe from concurrent modifications)
- Null elements allowed
- Collections.copy() can work with it

**Copy-On-Write Pattern Explained:**
- Normal write: System.arraycopy() creates new array
- New array containing modification
- After write: Reference updated to new array
- Old array garbage collected if no longer referenced
- Readers see either old or new array consistently

**Performance Characteristics:**

**Read Operations (O(1)):**
- Very fast
- No synchronization overhead
- Direct array access
- Multiple threads can read simultaneously

**Write Operations (O(n)):**
- Expensive
- Entire array copied
- New array created with modification
- Lock acquired during copy

**Example:**
```java
List<String> list = new CopyOnWriteArrayList<>();
list.add("A");
list.add("B");
list.add("C");

// Multiple reader threads can safely iterate
Thread reader = new Thread(() -> {
    for (String item : list) {
        System.out.println(item);  // No exception on modification
    }
});

// Writer thread
Thread writer = new Thread(() -> {
    list.add("D");  // Creates new array copy
});

reader.start();
writer.start();
```

**Iteration Example:**
```java
CopyOnWriteArrayList<Integer> list = new CopyOnWriteArrayList<>();
list.add(1);
list.add(2);
list.add(3);

// Safe from concurrent modifications
for (Integer num : list) {
    System.out.println(num);
    list.add(99);  // No ConcurrentModificationException
}
```

**Methods Comparison:**
---------------------------------------------------------------------------
| Operation        | ArrayList     | Vector    | CopyOnWriteArrayList     |
|------------------|---------------|-----------|--------------------------|
| Thread-safe      | No            | Yes       | Yes                      |
| Lock type        | None          | Full lock | Read: none, Write: lock  | 
| Read performance | O(1)          | O(1) slow | O(1) fast                |
| Write performance| O(1)          | O(1) slow | O(n)                     |
| Iterator         | fail-fast     | fail-fast | fail-safe                |
| Recommended      | Single-thread | Legacy    | Read-heavy multi-thread  |
---------------------------------------------------------------------------

**Copy Cost Analysis:**
```java
// When write happens:
String[] oldArray = ["A", "B", "C"];  // Original
String[] newArray = ["A", "B", "C", "D"];  // Copy + modification
// Old array discarded
```

**When to Use CopyOnWriteArrayList:**
- Read-heavy workloads (many readers, few writers)
- Event listener lists
- Snapshot iteration required
- Applications where consistency matters more than write speed

**When NOT to Use:**
- Write-heavy workloads (lots of modifications)
- Very large lists (copy overhead expensive)
- Memory-constrained situations (two copies in memory)

**Real-World Example - Event Listeners:**
```java
public class EventPublisher {
    private List<EventListener> listeners = new CopyOnWriteArrayList<>();
    
    public void addListener(EventListener listener) {
        listeners.add(listener);  // Occasional writes
    }
    
    public void publishEvent(Event event) {
        for (EventListener listener : listeners) {
            listener.onEvent(event);  // Frequent reads
        }
    }
}
```

**Advantages:**
- True thread-safety without synchronization
- Fast read operations
- Immutable snapshots for iteration
- No deadlock risk
- Fail-safe iterators

**Disadvantages:**
- Expensive write operations O(n)
- High memory usage (two arrays)
- Not suitable for write-heavy workloads
- Large copy overhead for big lists

24. # What is ConcurrentHashMap?

ConcurrentHashMap is a thread-safe Map implementation designed for high-concurrency multi-threaded applications.

**Definition:**
- Thread-safe alternative to HashMap
- Part of java.util.concurrent package
- Designed for concurrent read and write access
- Multiple threads can access different segments simultaneously
- Better performance than synchronized maps

**Key Characteristics:**
- Thread-safe without locking entire map
- Does NOT allow null keys or null values
- Multiple threads can read simultaneously
- Multiple threads can write to different buckets simultaneously
- Read operations are typically lock-free
- Iterator is fail-safe (safe from concurrent modifications)
- Good for high-concurrency scenarios

**Internal Locking Mechanism:**

**Java 7 and earlier (Segment-based):**
- Map divided into segments (default 16)
- Each segment has its own lock
- Up to 16 threads can write concurrently
- Reading doesn't require locking
- Lock granularity: Segment level

**Java 8+ (Node-level locks):**
- Uses node-level locks (bucket-level)
- Finer granularity than segments
- Only specific bucket locked during operation
- Multiple threads can write to different buckets
- Better concurrency than Java 7

**Performance Characteristics:**

| Operation        | HashMap | Hashtable   | ConcurrentHashMap |
|------------------|---------|-------------|-------------------|
| Thread-safe      | No      | Yes         | Yes               |
| Lock scope       | None    | Entire map  | Bucket/Segment    |
| Read concurrency | N/A     | 1 thread    | Multiple threads  |
| Write concurrency | N/A    | 1 thread    | Multiple threads (different buckets) |
| Null key         | Allowed | Not allowed | Not allowed       |
| Null value       | Allowed | Not allowed | Not allowed       |
| Performance      | Fastest | Slow        | Fast              |

**Example Usage:**
```java
Map<String, Integer> map = new ConcurrentHashMap<>();
map.put("A", 1);
map.put("B", 2);
map.put("C", 3);

// Multiple threads can read safely
Thread reader = new Thread(() -> {
    for (String key : map.keySet()) {
        System.out.println(key + " = " + map.get(key));
    }
});

// Multiple threads can write to different keys
Thread writer1 = new Thread(() -> map.put("D", 4));
Thread writer2 = new Thread(() -> map.put("E", 5));

reader.start();
writer1.start();
writer2.start();
```

**Important Methods:**

**Standard Methods:**
- get(K key): Get value (not locked)
- put(K key, V value): Add/update
- remove(K key): Remove entry
- containsKey(K key): Check existence
- containsValue(V value): Check if value exists
- clear(): Remove all entries

**Atomic Operations (Java 8+):**
- putIfAbsent(K, V): Atomic put if key not present
- remove(K, V): Remove only if key-value matches
- replace(K, V): Replace only if present
- forEach(BiConsumer): Safe iteration

**Example with Atomic Operations:**
```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// Atomic replace
map.put("counter", 0);
map.replace("counter", 0, 1);  // Only if value is 0

// Atomic put-if-absent
map.putIfAbsent("key", "value");  // Only if not present

// Safe iteration
map.forEach((key, value) -> {
    System.out.println(key + " = " + value);
});
```

**Iteration Safety:**
```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
map.put("A", 1);
map.put("B", 2);

// Iterator is fail-safe
for (String key : map.keySet()) {
    map.put("C", 3);  // Safe - doesn't throw exception
}
```

**Size Considerations:**
- size() returns approximate count (not exact under concurrency)
- May not include recent additions
- For approximate size use size()
- For exact size, accumulation required

**When to Use ConcurrentHashMap:**
- Multi-threaded read-heavy workloads
- Multiple threads accessing same map
- High-concurrency scenarios
- Server applications with many concurrent users
- Cache implementations
- Configuration maps accessed by multiple threads

**When NOT to Use:**
- Single-threaded applications (use HashMap)
- Need null keys/values (not supported)
- Need ordering (use ConcurrentSkipListMap)

**Comparison with Hashtable:**
```java
// OLD way - Hashtable (entire map locked)
Map<String, Integer> old = new Hashtable<>();

// NEW way - ConcurrentHashMap (segment locked)
Map<String, Integer> best = new ConcurrentHashMap<>();
```

**Advantages:**
- High concurrency with minimal locking
- Better performance than Hashtable
- Scalable for multi-core systems
- Fail-safe iterators
- Atomic operations available

**Disadvantages:**
- No null keys or values allowed
- Size query not always exact
- Slightly more memory than HashMap
- Not ordered

25. # Difference between HashSet and TreeSet?

**HashSet:**
- Unordered collection of unique elements
- No guaranteed iteration order
- Based on HashMap internally
- Uses hash code for storage location
- Not sorted collection
- Allows one null element
- Not synchronized (not thread-safe)
- Performance: O(1) average for add, remove, contains
- Suitable for membership testing
- Introduced in Java 1.2
- Better for performance-critical code
- Elements can be in any order

**TreeSet:**
- Ordered collection of unique elements
- Sorted in ascending order (by default)
- Based on Red-Black Tree (self-balancing BST)
- Uses Comparable/Comparator for ordering
- Sorted collection
- Does NOT allow null elements
- Not synchronized (not thread-safe)
- Performance: O(log n) for add, remove, contains
- Suitable for range queries and ordering
- Introduced in Java 1.2
- Better for ordered processing
- Elements always in sorted order

**Detailed Comparison Table:**

| Feature            | HashSet            | TreeSet                 |
|--------------------|--------------------|-------------------------|
| Order              | Unordered          | Sorted ascending        |
| Implementation     | HashMap            | Red-Black Tree          |
| Performance add    | O(1) average       | O(log n)                |
| Performance remove | O(1) average       | O(log n)                |
| Performance search | O(1) average       | O(log n)                |
| Null allowed       | One null           | No null                 |
| Synchronization    | Not synchronized   | Not synchronized        |
| Iteration order    | Unpredictable      | Ascending order         |
| Comparable needed  | No                 | Yes (default or custom) |
| Range queries      | No                 | Yes                     |
| Memory usage       | Less               | More (tree nodes)       |
| Use when           | Fast access needed | Ordering needed         |

**HashSet Example:**
```java
Set<String> hashSet = new HashSet<>();
hashSet.add("Charlie");
hashSet.add("Alice");
hashSet.add("Bob");
hashSet.add(null);

System.out.println(hashSet);  
// Output: [null, Bob, Charlie, Alice] OR
//         [Charlie, null, Alice, Bob] (unpredictable order)

// Iteration order varies (not guaranteed)
for (String name : hashSet) {
    System.out.println(name);
}
```

**TreeSet Example:**
```java
Set<String> treeSet = new TreeSet<>();
treeSet.add("Charlie");
treeSet.add("Alice");
treeSet.add("Bob");
// treeSet.add(null);  // Throws NullPointerException

System.out.println(treeSet);  
// Output: [Alice, Bob, Charlie] (always sorted)

// Iteration always in sorted order
for (String name : treeSet) {
    System.out.println(name);
}
// Output: Alice, Bob, Charlie
```

**Additional TreeSet Methods (SortedSet interface):**
```java
TreeSet<Integer> treeSet = new TreeSet<>();
treeSet.addAll(Arrays.asList(5, 1, 9, 3, 7));

// Range queries
treeSet.headSet(5);        // [1, 3] (less than 5)
treeSet.tailSet(5);        // [5, 7, 9] (greater than or equal to 5)
treeSet.subSet(3, 7);      // [3, 5] (3 <= x < 7)

// Navigation
treeSet.first();            // 1 (smallest element)
treeSet.last();             // 9 (largest element)
treeSet.floor(6);           // 5 (largest element <= 6)
treeSet.ceiling(6);         // 7 (smallest element >= 6)
```

**Custom Ordering with TreeSet:**
```java
// Default natural order (ascending)
Set<Integer> ascending = new TreeSet<>();
ascending.addAll(Arrays.asList(5, 1, 3));
System.out.println(ascending);  // [1, 3, 5]

// Custom order (descending)
Set<Integer> descending = new TreeSet<>((a, b) -> b - a);
descending.addAll(Arrays.asList(5, 1, 3));
System.out.println(descending);  // [5, 3, 1]
```

**LinkedHashSet - Insertion Order:**
```java
// Neither HashSet nor TreeSet
Set<String> linkedSet = new LinkedHashSet<>();
linkedSet.add("Charlie");
linkedSet.add("Alice");
linkedSet.add("Bob");

System.out.println(linkedSet);  
// Output: [Charlie, Alice, Bob] (insertion order preserved)
```

**Decision Guide:**

**Use HashSet when:**
- You need fastest access/search
- Order doesn't matter
- You have small number of elements
- Membership testing is priority
- You want O(1) performance

**Use TreeSet when:**
- Elements must be in sorted order
- You need range queries
- You need first/last element
- You want ordered iteration
- You can accept O(log n) performance

**Use LinkedHashSet when:**
- You need insertion order maintained
- You want O(1) performance like HashSet
- Order matters but not alphabetical

**Performance Summary:**
```            add      | remove   | search    |
---------------------------------------------------------------
HashSet:       O(1)     | O(1)     | O(1)      |(unordered, fastest)
LinkedHashSet: O(1)     | O(1)     | O(1)      |(insertion-ordered)
TreeSet:       O(log n) | O(log n) | O(log n)  | (sorted, slowest)
-----------------------------------------------------------------------              
```

**Thread-Safe Alternatives:**
- HashSet → Collections.synchronizedSet()
- TreeSet → Collections.synchronizedSortedSet()
- (Better) → ConcurrentSkipListSet for TreeSet-like behavior
