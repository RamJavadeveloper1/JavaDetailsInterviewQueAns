# 🔥 5️⃣ How does @Transactional work internally?

### ✅ Senior-Level Answer:

* Spring uses AOP proxy mechanism.
* It creates proxy around bean.
* When transactional method is called → proxy intercepts.
* It opens transaction.
* Executes method.

* If RuntimeException → rollback.

Important:

* Works only on public methods.
* Self-invocation does not work.

# 🔥 Question:

### How does `@Transactional` behave in case of checked exception?

---

# ✅ Correct Interview Answer (7+ Years Level)

By default:

* Spring rolls back **only on unchecked exceptions**
  (RuntimeException and Error)

* It does **NOT roll back on checked exceptions**

Example:

```java
@Transactional
public void process() throws Exception {
    saveData();
    throw new Exception("Checked exception");
}
```

👉 Transaction will NOT rollback by default.

Data will be committed.

---

# 🔥 Why?

Because Spring transaction management is designed to rollback on:

* RuntimeException
* Error

Checked exceptions are considered business exceptions, so commit happens.

---

# 🔥 How to Rollback for Checked Exception?

We must specify:

```java
@Transactional(rollbackFor = Exception.class)
```

or

```java
@Transactional(rollbackFor = {CustomCheckedException.class})
```

Now transaction will rollback.

---

# 🎯 Interview Tip (Important)

After answering, add:

> "This behavior is configurable using rollbackFor and noRollbackFor attributes."

This makes you look senior.

---

# 🧠 Bonus Senior-Level Knowledge

If you catch the exception inside method and don’t rethrow:

Transaction will commit.

Example:

```java
@Transactional
public void test() {
   try {
       saveData();
       throw new RuntimeException();
   } catch(Exception e) {
       // swallowed
   }
}
```

👉 No rollback happens because exception never reaches proxy.

Spring uses **AOP proxy mechanism** to implement `@Transactional`.

When a method annotated with `@Transactional` is called:

* Proxy opens transaction
* Executes method
* Commits or rollbacks

However:

In self-invocation (one method inside same class calling another transactional method):

```java
public class OrderService {

    public void methodA() {
        methodB(); // self invocation
    }

    @Transactional
    public void methodB() {
        // transactional logic
    }
}
```

Here:

* `methodA()` calls `methodB()` directly
* Call does NOT go through proxy
* It bypasses Spring AOP proxy
* Therefore transaction is NOT applied

---

# 🔥 Why Exactly?

Because Spring creates a **proxy around the bean**, not inside the bean.

Internal method calls use `this.method()` reference.

Proxy is skipped.

---

1. Move method to another bean
2. Inject same bean using proxy reference
3. Use AspectJ instead of proxy-based AOP

Example:

```java
@Autowired
private OrderService orderService;

orderService.methodB();
```

Now it goes through proxy.

# 🔥 Next Question (Even More Senior)

# What is the difference between:

* Isolation Level READ_COMMITTED
* Isolation Level REPEATABLE_READ
* Isolation Level SERIALIZABLE

Very good. You said honestly you don’t know clearly.
Now I will explain this properly — this is **VERY IMPORTANT for 7+ years interviews**.

Isolation levels control **how transactions see data from other transactions**.

They solve problems like:

* Dirty read
* Non-repeatable read
* Phantom read

Let’s go step by step.

---

# 🔥 First Understand 3 Problems

## 1️⃣ Dirty Read

Transaction A reads data that Transaction B updated but NOT committed yet.

If B rolls back → A read invalid data.

---

## 2️⃣ Non-Repeatable Read

Transaction A reads same row twice.
Between reads, Transaction B updates and commits.
A sees different value second time.

---

## 3️⃣ Phantom Read

Transaction A reads rows using condition (e.g., salary > 50000).
Transaction B inserts new matching row and commits.
A re-runs query → extra row appears.

---

Now isolation levels:

---

# 🔥 1️⃣ READ_COMMITTED

👉 Can only read committed data.
👉 Prevents Dirty Read.
👉 Allows Non-repeatable Read.
👉 Allows Phantom Read.

### Example:

Transaction A:

```sql
SELECT salary FROM employee WHERE id=1;
```

Transaction B:

```sql
UPDATE employee SET salary=70000 WHERE id=1;
COMMIT;
```

Transaction A reads again → salary changed.

This is allowed.

---

# 🔥 2️⃣ REPEATABLE_READ

👉 Prevents Dirty Read
👉 Prevents Non-repeatable Read
👉 Allows Phantom Read (in theory)

Meaning:
If A reads row once, it will always see same value during transaction.

Even if B updates and commits, A still sees old value.

(MySQL actually prevents phantom too using MVCC.)

---

# 🔥 3️⃣ SERIALIZABLE

👉 Highest isolation level
👉 Prevents Dirty Read
👉 Prevents Non-repeatable Read
👉 Prevents Phantom Read

Acts like transactions run one by one.

But:
❗ Very slow
❗ Reduces concurrency

---

# 🔥 Quick Comparison Table

| Isolation Level | Dirty Read  | Non-Repeatable | Phantom     |
| --------------- | ----------- | -------------- | ----------- |
| READ_COMMITTED  | ❌ Prevented | ❌ Allowed      | ❌ Allowed   |
| REPEATABLE_READ | ❌ Prevented | ❌ Prevented    | ❌ Allowed   |
| SERIALIZABLE    | ❌ Prevented | ❌ Prevented    | ❌ Prevented |

---

# 🔥 Interview Power Answer

If interviewer asks:

You say:

> READ_COMMITTED prevents dirty reads but allows non-repeatable reads and phantom reads.
> REPEATABLE_READ ensures row-level consistency but still allows phantom reads.
> SERIALIZABLE provides full isolation by executing transactions as if serially.

That’s clean.

---

# 🔥 Important for You (Spring)

Default isolation level depends on DB:

* MySQL → REPEATABLE_READ
* Postgres → READ_COMMITTED

In Spring:

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
```
