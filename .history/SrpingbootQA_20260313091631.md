# 🔥 Question:

### How does `@Transactional` behave in case of checked exception?

---

# ✅ Correct Interview Answer (7+ Years Level)

By default:

* Spring rolls back **only on unchecked exceptions**
  (RuntimeException and Error)

* It does **NOT roll back on checked exceptions**

Example:

```java
@Transactional
public void process() throws Exception {
    saveData();
    throw new Exception("Checked exception");
}
```

👉 Transaction will NOT rollback by default.

Data will be committed.

---

# 🔥 Why?

Because Spring transaction management is designed to rollback on:

* RuntimeException
* Error

Checked exceptions are considered business exceptions, so commit happens.

---

# 🔥 How to Rollback for Checked Exception?

We must specify:

```java
@Transactional(rollbackFor = Exception.class)
```

or

```java
@Transactional(rollbackFor = {CustomCheckedException.class})
```

Now transaction will rollback.

---

# 🎯 Interview Tip (Important)

After answering, add:

> "This behavior is configurable using rollbackFor and noRollbackFor attributes."

This makes you look senior.

---

# 🧠 Bonus Senior-Level Knowledge

If you catch the exception inside method and don’t rethrow:

Transaction will commit.

Example:

```java
@Transactional
public void test() {
   try {
       saveData();
       throw new RuntimeException();
   } catch(Exception e) {
       // swallowed
   }
}
```

👉 No rollback happens because exception never reaches proxy.

Spring uses **AOP proxy mechanism** to implement `@Transactional`.

* Executes method
* Commits or rollbacks

public class OrderService {

* Call does NOT go through proxy
* It bypasses Spring AOP proxy

Internal method calls use `this.method()` reference.

Proxy is skipped.

1. Move method to another bean
2. Inject same bean using proxy reference
3. Use AspectJ instead of proxy-based AOP

Example:

```java
@Autowired
private OrderService orderService;

orderService.methodB();
```

Now it goes through proxy.

* saveOrder() rolls back

Because logAudit ran in separate transaction.

# What is the difference between:

* Isolation Level READ_COMMITTED
* Isolation Level REPEATABLE_READ
* Isolation Level SERIALIZABLE

Very good. You said honestly you don’t know clearly.
Now I will explain this properly — this is **VERY IMPORTANT for 7+ years interviews**.

Isolation levels control **how transactions see data from other transactions**.

They solve problems like:

* Dirty read
* Non-repeatable read
* Phantom read

Let’s go step by step.

---

# 🔥 First Understand 3 Problems

## 1️⃣ Dirty Read

Transaction A reads data that Transaction B updated but NOT committed yet.

If B rolls back → A read invalid data.

---

## 2️⃣ Non-Repeatable Read

Transaction A reads same row twice.
Between reads, Transaction B updates and commits.
A sees different value second time.

---

## 3️⃣ Phantom Read

Transaction A reads rows using condition (e.g., salary > 50000).
Transaction B inserts new matching row and commits.
A re-runs query → extra row appears.

---

Now isolation levels:

---

# 🔥 1️⃣ READ_COMMITTED

👉 Can only read committed data.
👉 Prevents Dirty Read.
👉 Allows Non-repeatable Read.
👉 Allows Phantom Read.

### Example:

Transaction A:

```sql
SELECT salary FROM employee WHERE id=1;
```

Transaction B:

```sql
UPDATE employee SET salary=70000 WHERE id=1;
COMMIT;
```

Transaction A reads again → salary changed.

This is allowed.

---

# 🔥 2️⃣ REPEATABLE_READ

👉 Prevents Dirty Read
👉 Prevents Non-repeatable Read
👉 Allows Phantom Read (in theory)

Meaning:
If A reads row once, it will always see same value during transaction.

Even if B updates and commits, A still sees old value.

(MySQL actually prevents phantom too using MVCC.)

---

# 🔥 3️⃣ SERIALIZABLE

👉 Highest isolation level
👉 Prevents Dirty Read
👉 Prevents Non-repeatable Read
👉 Prevents Phantom Read

Acts like transactions run one by one.

But:
❗ Very slow
❗ Reduces concurrency

---

# 🔥 Quick Comparison Table

| Isolation Level | Dirty Read  | Non-Repeatable | Phantom     |
| --------------- | ----------- | -------------- | ----------- |
| READ_COMMITTED  | ❌ Prevented | ❌ Allowed      | ❌ Allowed   |
| REPEATABLE_READ | ❌ Prevented | ❌ Prevented    | ❌ Allowed   |
| SERIALIZABLE    | ❌ Prevented | ❌ Prevented    | ❌ Prevented |

---

# 🔥 Interview Power Answer

If interviewer asks:

You say:

> READ_COMMITTED prevents dirty reads but allows non-repeatable reads and phantom reads.
> REPEATABLE_READ ensures row-level consistency but still allows phantom reads.
> SERIALIZABLE provides full isolation by executing transactions as if serially.

That’s clean.

---

# 🔥 Important for You (Spring)

Default isolation level depends on DB:

* MySQL → REPEATABLE_READ
* Postgres → READ_COMMITTED

In Spring:

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
```

# 1️⃣ Why Constructor Injection Makes Dependency Mandatory

Example:

```java
public class OrderService {

    private PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```

Here:

When Spring creates the object:

```
new OrderService(paymentService)
```

If `paymentService` is missing → object **cannot be created**.

So dependency becomes **required**.

---

### Setter Injection Case

```java
public class OrderService {

    private PaymentService paymentService;

    public void setPaymentService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```

Object can be created like this:

```
new OrderService()
```

But dependency may **never be set**.

So object exists in **invalid state**.

That is why constructor injection ensures **mandatory dependency**.

---

Example:

```java
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```

Notice:

```
private final PaymentService paymentService;
```

`final` means:

* Value assigned once
* Cannot change later

So dependency **never changes after object creation**.

That makes class **immutable**.

---

### Setter Injection Problem

```java
orderService.setPaymentService(service1);
orderService.setPaymentService(service2);
```

Dependency can change anytime.

Not immutable.

---

# 3️⃣ Why Constructor Injection Improves Testability

Testing example:

With constructor injection:

```java
PaymentService mock = new FakePaymentService();
OrderService orderService = new OrderService(mock);
```

You can easily pass **mock object**.

Very clean.

---

### Field Injection Problem

```java
@Autowired
private PaymentService paymentService;
```

You cannot set dependency easily in unit test.

You must use:

* Reflection
* Spring Test context
* Mockito injection

Which makes testing harder.

| Feature              | Constructor | Setter    | Field  |
| -------------------- | ----------- | --------- | ------ |
| Mandatory dependency | ✅ Yes       | ❌ No      | ❌ No   |
| Immutability         | ✅ Yes       | ❌ No      | ❌ No   |
| Easy testing         | ✅ Yes       | ⚠ Partial | ❌ Hard |

# ❓ Difference between `@Component`, `@Bean`, and `@Configuration`

These are all related to **Spring Bean creation**, but they work differently.

---

# 1️⃣ `@Component`

`@Component` tells Spring:

👉 "Create this class as a Spring bean automatically."

Spring finds it using **Component Scanning**.

Example:

```java
@Component
public class PaymentService {
}
```

Spring will automatically create a bean for this class.

Other similar annotations are:

* `@Service`
* `@Repository`
* `@Controller`

These are specializations of `@Component`.

Example:

```java
@Service
public class OrderService {
}
```

---

# 2️⃣ `@Bean`

`@Bean` is used when **we want to manually create a bean inside a configuration class**.

Example:

```java
@Configuration
public class AppConfig {

    @Bean
    public PaymentService paymentService() {
        return new PaymentService();
    }
}
```

Here:

Spring will create a bean using this method.

---

### When do we use `@Bean`?

When we cannot modify the class.

Example:
Third-party library class.

Example:

```java
@Bean
public ObjectMapper objectMapper() {
    return new ObjectMapper();
}
```

---

# 3️⃣ `@Configuration`

`@Configuration` tells Spring:

👉 This class contains **bean definitions**.

Example:

```java
@Configuration
public class AppConfig {
}
```

Inside this class we usually write `@Bean` methods.

---

# 🔥 Important Internal Behavior

`@Configuration` classes are enhanced by **CGLIB proxy**.

This ensures that:

Even if one bean method calls another bean method, **Spring returns the same singleton bean**.

Example:

```java
@Configuration
public class AppConfig {

    @Bean
    public A a() {
        return new A(b());
    }

    @Bean
    public B b() {
        return new B();
    }
}
```

Spring ensures **B is created only once**.

| `@Configuration` | Class that contains bean definitions               |

# ❓ Why does `@Transactional` only work on **public methods**?

### Short Answer (Interview Ready)

`@Transactional` works using **Spring AOP proxies**, and proxies intercept **external method calls**.
Since private, protected, and package-private methods cannot be accessed through the proxy, the transaction advice is applied only to **public methods**.

---

# 🔹 How `@Transactional` Works Internally

Spring implements transactions using **AOP (Aspect Oriented Programming)**.

When a class has `@Transactional`, Spring creates a **proxy object** around that bean.

Example:

```java
@Service
public class OrderService {

    @Transactional
    public void createOrder() {
        saveOrder();
    }
}
```

Spring actually creates something like:

```
Client → Proxy → OrderService
```

The proxy:

1. Opens the transaction
2. Calls the actual method
3. Commits or rolls back transaction

---

# 🔹 Why Only Public Methods?

Spring AOP uses **proxy-based method interception**.

Proxy can intercept only **public methods that are called from outside the bean**.

If the method is:

* `private`
* `protected`
* `default`

Spring cannot intercept it through proxy.

Example:

```java
@Transactional
private void processOrder() {
}
```

This **will not work**.

Because proxy cannot call a private method.

---

# 🔹 Example Problem

```java
@Service
public class OrderService {

    public void createOrder() {
        processPayment();   // self invocation
    }

    @Transactional
    public void processPayment() {
    }
}
```

Here:

`createOrder()` calls `processPayment()` directly.

Call does **not go through proxy**, so transaction will **not work**.

This is called **self-invocation problem**.

---

# 🔥 Interview Power Statement

You can say:

> Spring implements `@Transactional` using proxy-based AOP. Since proxies intercept external public method calls, transactional behavior is applied only to public methods. Private or internal method calls bypass the proxy and therefore do not trigger transactions.

---

# ⭐ Senior-Level Extra Knowledge

There are **two types of Spring AOP proxies**:

1️⃣ **JDK Dynamic Proxy**

* Used when class implements interface

2️⃣ **CGLIB Proxy**

* Used when no interface exists

CGLIB creates a subclass at runtime.

---

# ❓ If a `@Transactional` method calls another `@Transactional` method inside the same class, will transaction work?

### ❌ Answer

No, it will **not work as expected**.

---

# 🔹 Reason

Spring transactions work using **proxy-based AOP**.

The proxy intercepts **external method calls**.

Example:

```
Client → Proxy → Service Method
```

But when a method inside the same class calls another method:

```
methodA() → methodB()
```

The call happens directly using `this.methodB()`.

It **does not go through the proxy**, so Spring cannot apply transactional logic.

This is called **self-invocation problem**.

---

# 🔹 Example

```java
@Service
public class OrderService {

    @Transactional
    public void methodA() {
        methodB();  // direct call
    }

    @Transactional
    public void methodB() {
        // transaction expected
    }
}
```

Here:

* `methodB()` will **not start a new transaction**
* Because the call bypasses the proxy.

---

# 🔹 How to Fix It

### Option 1 (Most Common)

Move the second method to another service.

```java
@Service
public class PaymentService {

    @Transactional
    public void methodB() {
    }
}
```

Then inject it.

---

### Option 2

Inject the proxy of the same bean.

```java
@Autowired
private OrderService orderService;

orderService.methodB();
```

Now call goes through proxy.

You can say:

> Self-invocation bypasses the Spring AOP proxy, so transactional advice is not applied.

# 1️⃣ What is the difference between JDK Proxy and CGLIB Proxy?

### JDK Dynamic Proxy

* Used when the class **implements an interface**
* Proxy created for **interfaces**
* Uses Java `java.lang.reflect.Proxy`

Example:

```java
public interface PaymentService {
    void pay();
}
```

Spring creates proxy based on the interface.

---

### CGLIB Proxy

* Used when class **does not implement interface**
* Creates **subclass of the class**
* Uses **bytecode generation**

Example:

```java
public class PaymentService {
    public void pay() {}
}
```

Spring creates subclass dynamically.

---

### Quick Difference

| JDK Proxy             | CGLIB Proxy              |
| --------------------- | ------------------------ |
| Works with interfaces | Works with classes       |
| Uses Java reflection  | Uses bytecode generation |
| Faster                | Slightly heavier         |
